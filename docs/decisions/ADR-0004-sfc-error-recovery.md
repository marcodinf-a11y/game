# ADR-0004: Log-and-Ask SFC Error Recovery with Deterministic Replay

**Status:** Accepted

## Context

The simulation runs `Ledger.CheckConsistency()` every tick (NFR-DTA-001). If an SFC imbalance is detected, neither the PRD nor the Architecture specified what happens next. Options: halt, continue silently, roll back, throw.

An SFC failure cannot be caused through the game's typed interface — every transaction is double-entry, generated by the engine's own code. The only two causes are:

1. An undiscovered bug in the simulation code.
2. External memory manipulation (e.g., Cheat Engine).

## Decision

Adopt a **log-and-ask** strategy with deterministic replay for recovery:

1. `CheckConsistency()` fails — log full diagnostics (which accounts, what imbalance, which tick, expected vs. actual totals).
2. Pause the simulation and show a dialog: *"Accounting inconsistency detected at tick N. This is either a bug (please report it with the log file) or caused by external memory modification. You can continue with the inconsistent state, or restore to the last consistent state."*
3. **Continue** — unpause and keep ticking with the broken state. Covers the Cheat Engine case.
4. **Restore** — replay the simulation from tick 0 to tick N-1 using the policy input log and the original seed. The simulation is deterministic (seeded `IRandom`), so replaying with the same inputs reproduces the exact state at any prior tick.

## Consequences

- No snapshot or serialization infrastructure needed for recovery — replay reconstructs state.
- The Game Controller must record a policy input log: a list of `(tick, command)` pairs.
- Replay-based restore reuses the same deterministic replay infrastructure planned for post-MVP save/load.
- `ReplayToTick(int targetTick)` method added to `ISimulationFactory`, taking the original seed and a policy input log.
- Replaying without rendering is fast, so even long games restore in acceptable time.
- The player is never surprised by a silent halt or crash — they choose what happens.
